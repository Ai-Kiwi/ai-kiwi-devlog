## Storing Process Data

First off storing processes running, originally my through when I was brand new to this os was to code it with a kernel process ID and global ID system and then convert global ID to kernel ID the reasoning for this was that it allowed me to have one large array of kernel processes greatly simplifying things, since then we now have memory management working so we can do a better way. The idea was that the Global process ID would point to the kernel process. This would limit how many processes can run, but would mean it can be stored in one array much more easily, and the process ID can be counted up, allowing the process ID to keep counting up, lowering process ID collisions (process died and a new one quickly replaces it with the same ID).

I ended up scrapping this idea because there was a much simpler one: radix tables. Originally, I considered this idea but dismissed it because of how many RAM fetches it would require to follow the path; however, it can surprisingly be relatively low-cost if you merge different bit sections into a single address point. So, instead of each RAM fetch having a 0 or a 1 for the 2 next two paths you can take, you store an array of locations you could take, and then you know which to pick in this array based on the value in the section of the key you are reading. 

It did cross my mind to store all the processes as a list of addresses, since that would be fast in the cache, but it would cost way too much, and even though it's fast in the cache, this is a huge amount of data at this scale. For example, with the next, which goes up to 65535, this would mean 524280 bytes in just an index table (524.28 kilobytes). Which, in the much more standard and useful measurement, is 131070 nibbles. This is a huge amount of data that wouldn't fit in cache and would take ages to iterate over. The cost would just be much higher.

I won't go into details about coding the radix table here, as another devlog is focused on that.d But the idea was to code a generic function that can be reused. As I was planning it, I could see many uses for places where it could come in handy, such as IPC data, file handles, or page table data (not all page tables, only interesting ones like those linked to a process or file cache).

For picking the size, I originally was thinking a byte per table. As fewer fetches are better, and it would only be 2 fetches on a likely 16-bit size, I was planning. However, this results in 2048 bytes per table, which is a large amount. The better amount I ended up going for is a nibble per page, which is 4 bits. This lands at 128 bytes per page, which is much better, but it requires more fetches to retrieve the data. However, if there are fewer processes or processes are more spread out, it is less wasteful. I will admit the numbers here aren't completely my own; however, I do agree with the end conclusion. 

For the time being, when a table is free and there are no processes on any of its leaves, it will be left alone. However, in future, it will be trimmed. I am not completely sure how to do this without killing performance, as there is an advantage to having it ready for when other processes are slotted in place. So for the time being, it will be left and possibly revisited later. An idea I had was to store the number of children. But this does raise issues with that one extra, putting it into another allocator size bracket, resulting in a large amount of wasted space. It would also be another value to edit and place, which could lead to an error.

When a process is freed my current idea is that It will just have its data left and marked as dead and can then be overriden however I am not sure if this is better or being able to loop over the radix looking for an empty item is better, likely being able to loop over radix will be better as it also lets there be generic radix function for it so I will end up going for that approch.

## Coding Process Data

The next thing I coded was a system to get a blank process. This would find a free process ID and then bind the process data to the radix created for processes. Currently, the approach is a naive loop over each process and checks whether it has data. This means more fetches for testing if free, if there are a lot in use, or if it takes effort to find. These also aren't quick fetches, as they're large amounts of RAM fetches for each and not sequential. This means it might be better to store a blank pointer in vma. Since the process is dead, it would be faster to loop over and use a radix function to find the next blank process. But it would also very likely return the ID of the process just freed, rather than iterate up. This could be fixed by not relying on data for already used processes twice, but there could potentially be security issues here, and this is another place for an exploit. A general rule in OS dev is to stack as many layers of security on top of each other as possible.  

The next thing I wanted to look into was regarding start locations, as most programs aren't placed directly into the start location. If an invalid pointer occurs, it will point to 0x0, triggering an exception and crashing the program, preventing errors from getting out of hand. While looking around for this, I found out from researching that a lot of this info, such as stack location, libraries to load, BSS location, and start location, etc., is not actually decided by the process when it starts, but actually coded into the ELF file format. I could go about skipping this step for now and just getting away with dumping the process data into RAM via VMA; however, I felt doing everything right to start with was better than trying to skip ahead and pay later for a shaky foundation and spend longer swapping everything out, so onto coding an ELF parser.

## ELF parser

I was actually wondering while coding this why even use a ELF parser and from looking into it and reasoning for why it is used, it turns out you can completly get away without one, however ELF allows you to much more simply load all of the data instead of having to mess around loading everything in as the process is running and have the process ask for stuff and figure out locations. ELF is also quite a clean format; it has a lot of room for expanding and encodes all the info you would reasonably need, which is why most operating systems use it, even a fair amount of proprietary ones.

When reading ELF data, it likely won't be loaded as a straight line. It will be split into different pages of data. This means that a different approach will have to be taken on this front, where instead of reading as one continuous whole, we have to read as multiple different splits. These will be pages that I will break up into, as it's what most of the rest of the code is going to use. This means that there needs to be a way to convert the relative location ot the page location and then the location on that page. To achieve this, I choose to use radix tables. A smart trick here is to fetch only the radix table root; depending on the tree's size, you can then decide which level up the tree to pick as the root. Meaning you can do fewer RAM fetches if the data is earlier. This could also use bit tricks so you can find which page it is without any if statements. The benefit of this is that it means faster fetching.

Some ELF spesfic information, I choose to hardcode 64bit locations as this OS Is already fairly hard coded for 64bit (will refuse load if not 64bit), choose to ignore handling exstnetsions as the way risc-v is designed it will just trap when a invalid instruction happens, later can be repalced with a error message on running program warning you and then even more later can use software emulation to make it work after warning you.

## VMA Design

In a process, dynamically loaded libraries will be marked as read-only. If a process attempts to make an edit, it will fire a trap. After this trap is fired, it can be seen that it was attempting to edit a library, so a 2nd copy can be made, which is the library but with editing permissions. The idea is that this 2nd copy wouldn't override the old one, so each process gets its own lib state without loading the lib for every process. This could be expensive each time, but from what I understand and what I've heard, it's fairly rare for a lib to do this, so it's not a huge cost. This is the common approach.

> [!NOTE]
> ### Note About IPC
> Another problem that I realised here is that you can't know if a lib or if the process itself is calling a function, or it would be very difficult to do (can see ways, but hacky and complex). This means the original IPC system I had in mind, where a lib creates a Caster, for example, wouldn't work, as it would appear as the process using the lib created it. The fix I decided on was that you would still be allowed to create them; however, they would be limited to the libraries you are attached to or to yourself. Then, permissions can determine who can use the library. The ironic part with this is that I originally planned not to have perms, and then they are a vital part of IPC security. This also comes with the downside of now being a very aggressive perm system, so there would be a lot of perm management. I can see a few other problems with this, like how the kernel handles who can access its Casters and whatnot, but I feel this problem is far enough away that it would be best to leave it for now and revisit it at a later date.

After the elf is passed, there needs to be a way to actually store the data os if a page is ready or assigned, you are able to know what info should be stored in that section. To achieve this, I thought it would be best to keep a list of sections for each process. There would also be a map for VMA to load file data pages. The page data that is pointed to wouldn't be the page itself, but a page data struct which stores info like page number, who has access, etc.

For actually storing this info, I plan to use ranges. One big list with start position and size. And when finding you, loop until you are within range of one of them, then use the info for that for section info. This means you can find what section applies to a location quickly. There is an idea for finding in this list of using binary search, which would, in theory, be faster; however, from my knowledge, RAM has a minimum amount of data it can normally collect in one go. This means, in practice, most fetches from RAM to CPU, depending on the architecture, are at least 64 bytes, meaning if you fetch in the middle, it's 2 fetches compared to just fetching at the start, where it fetches everything in one go. This means the approach I will take is to just loop from the start to the end, looking for the closest one. By storing the start, end and location of the section info on the page in one go, it means it is quick to look for which section you are inside of. Worth noting for this is that it will be sorted by start address, so early exit can be done instead of having to loop over all of them. A porntail issue/worth noting here is that sections can be overlapping from the ELF file.

For storing the pages that have actually been mapped to the processors, there will be a radix table that stores information. The table will not point to the page itself, but instead the info for the page. The reason for this is that it has to store info about that page, like what it's used for and who's using it. Worth noting, this means there will be 2 different tables for a similar thing, the internal VMA mapping table and my own internal one. However, my system allows for a lot more info to be added and to make it more generic, so it will be used, and the internal arch-specific one will just be updated as info is changed. An example of this is if a VMA is for a shared lib it will point to a page info which will define it as being a lib, it will also define which processes are using it as well. The idea I came up with is that if the process is killed, it can go to all these pages to see if it no longer has anything using it, and decide whether to clear it out. In terms of how this is mapped, it will be treated as a file cache, but can't be dropped as it is "opened" by some processes. However, this raises an issue if the process edits it. To handle this, I am planning to create a new copy on edit. This new copy will not be a file cache but just RAM data assigned to the process. This means you don't get 2 copies for open process data, making it cleaner.

The approach used here with radix also brings up the idea of gigapages with page structure data, which means you could store info like how many pages are stored, this will be left out, but in future, it can be added. A clean way to do it is in the page data structure store how many sequential pages it is, so it can store the start location and how many pages after that it applies to, then just go over each of them to free. I did consider adding to the page allocator, but this would require storing info for each page, like how many pages long it is, and info like that, and would require fetching for each time you free or request a page. This would also have to rely on the allocator, meaning circular dependencies. For this reason, I decided to leave it out for complexity and added cost for not a lot of benefit, the idea is that the process itself that asks for pages would free the amount they requested themselves. The page data idea talked about before is an optional add-on not needed, so if the kernel wants a page, it just gets one; it doesn't have to try to ask for one and then rent it with parms and stuff. This would only be a system for process pages to talk about why they own a page.

As a better fix to this approach later, I am thinking of coding another allocator type that supports huge multipage assignments, but this will be a feature for later. Likely will later do an "allocator extended" devlog or something along those lines when I do it.

## Filesystem

Something worth mentioning is that, depending on your drive, the sizes it returns in data can be different. For example, you may ask for 3 bytes from the disk, but it will not matter what it returns, the minimum amount (for example, 512 bytes). Because of this reason and also for performance, the best thing to do is cache the data. My thought here is to reuse the same size as the internal pages of 4096B to greatly simplify everything and reduce the risk of errors. From my understanding, afterwards, this is the same idea most other OS use in practise. It also allows for the data to slot into pages without any cost, as without it would be wasting some, or you would have to store which half or quarter it is, which would also be wasteful, as there's more info to store.

Most ideas take this idea further, and they look if you read right near the edge and use this to assume you're fetching a lot of data sequentially, so it will trigger prefetch for data afterwards. OS like Linux are very complex in how they predict prefetching files and clearing out of cache, I will likely do a simple approach similar to how Linux does it. 

My original idea was only to expose page pointers for processes to use however, I quickly realised this is a bad idea and diverting the problem and the much better way to handle it is to instead have a useful api like read range and read u64, etc., as this should be the goal of the fgile system driver to handle reading everything and which pages to use and what not. 

I was originally planning to have the data point to a page ID, then have the page system return the actual page location from that. However, I later decided it would be better to just have a generic page data structure, which is used for different use cases, rather than just having them all point to the page location. The reasoning here is that instead of going page > page data, you would always start at page data, meaning you can also cheaply move the page if it's needed to. It makes some things more complex, like finding all page cache files; however, it also makes other things simpler.

For reading files, I am planning to do an async api that returns an ID, which can be used to see if it's able to be read or what needs to be done for it to be free. I might add a sync API; however, you should never really use sync in kernel space, as it wastes time doing nothing while it fetches data, not to mention making things messy and holding up info. Or, throughout this, while reading, ' async 'is the wrong word to use, as it is sync; it would yield, so really I will do a sync api, but it will yield to async when the response is ready.

An interesting thing I found related to reading and writing is that you only have a single interrupt that is given for read or write finishes. This means that to know what is reading or writing, you have to keep track of yourself in a list. It also can't know how many reads or writes have finished, as there is only 1 bit in the register for firing interrupts, meaning that 2 reads or writes finish before you handle the interrupt, and only one is fired. This means we have to keep a list of things to read or write and work our way through that list. Likely with a circular ring buffer, however, there would be real issues if you go over this limit, so not quite sure on the approach yet, could do a linked list, but I can imagine there would be a performance cost. NVMe tells you which has finished reading and writing, and keeps a list of them. I haven't looked into the specifics of it yet, but it is good to know. 

For the FS reading data I am thinking to do radix table of location to page used, for performance here I can make it so it will store each levels most left entry, idea is that if you want to store something but it is in first radix tree level it can just go directly to that tree level greatly speeding up fetching or if in 2nd can go to 2nd level. This means it can have high speed and low RAM fetches in small files, but in huge files, it still works with the same approach. Because of the exponential nature of radix tables, it also means the approach would work surprisingly well.

My idea here was to use a global ID for files and a similar idea for folders. It would greatly simplify storing all this data, and when stuff relies on it. From what I have heard, this idea is used both in real OS and also is used internally for many filesystem formats, which makes a lot of sense as it's much better than alternatives like the filesystem using full paths as ID. I do see issues here with global ID conflicts between disks; however, I feel, for the time being, I will just use a global ID for each file on the OS internally, then later, when issues arise with this system, add some way for deduplication. I can see the approach of having an internal file ID different from a disk file ID and having internal save where and which ID for the real file, which is likely an approach that I will later take.
