# Move to hashmaps
Actually, after taking the opportunity to learn how HashMaps really work in an unrelated area, I decided to look into them for this operating system project. I found out they do outperform and are actually a better idea in some areas. The nature of a HashMap means that if there is a small number of items, you are able to fetch whatever data in 1 operation, and if there are more, it is able to sort itself out. However, they do have some side effects, such as resizing, a full rebuild, and allocation. 
There are multicore issues here, as we are doing a lock for each entry. You can further push this to a read/write lock for each entry, meaning you don't lock it and stop others if you are just reading. 

As for which one to use, surprisingly, a linked-list bucket for each entry is the common approach. The idea being you have a list of items for each, with each pointing to the others. This sounds like a bad idea; however, if there aren't many duplicates, it's fine in practice, and the cost isn't that huge for the occasional request.

As far as sizing goes, typically, hashmaps need to be at most 75% full, as past that it gets exponentially worse, not linear. Linux uses your RAM amount and scales it to a relative size that is a good fit. The reason is that resizing them is very, very expensive, and it's cheaper to just reserve a large amount that's unlikely to be overrun, as is the case. It sounds silly, but the amount of data you can store in a HashMap per size is quite large. At one item per 8 bytes (64-bit RAM address), you end up with 1MB, which is 125k items, which is a huge amount. 

As for picking the item it will be a random number generation from the key, this prevents overlaps and spreads it out. A intressting note here is that you also need to make sure you use a random key on boot for the seed of it as a potential security issue exisists here with finding keys which all go to the same location and spamming them, for example on a server you could create files or request with the same key and have themm all go in a large linked list which would take a large amount of time to find the location you are looking for. 

Interesting note regarding the key algorithm is that typically when they recommend you to use prime numbers for the size of hashmaps (or at least was when I learnt it), the reason is that you don't want 2 values to end up with the same location and having prime numbers means you don't get multiples, so it makes it less likely. However, the stronger your randomisation algorithm actually is, the less this matters, so if you are able to make random enough hashes from the keys, having a prime-sized hashmap doesn't really matter. For that reason, itis okay to do it to a power of 2, which is what I'm planning to do. Also, the cost of finding a prime at this size would be insane and very wasteful. 

Depending on your situation, whether to use this or not. For sequential work, a radix tree is best, but for random items, a HashMap is best. For this reason, a page cache per file works best as a radix, but an ID for files or handles works best as a hashmap.
