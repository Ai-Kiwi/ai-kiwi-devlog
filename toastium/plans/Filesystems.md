## Filesystem

Something worth mentioning is that, depending on your drive, the sizes it returns in the data can vary. For example, you may ask for 3 bytes from the disk, but it will return the minimum amount (e.g., 512 bytes). For this reason, and for performance, the best approach is to cache the data. My thought here is to reuse the same size as the internal pages, 4096B, to greatly simplify everything and reduce the risk of errors. From my understanding, afterwards, this is the same idea most other OS use in practise. It also allows the data to slot into pages at no cost; without it, you'd waste some, or you would have to store which half or quarter it is, which would also be wasteful since there's more info to store.

Most operating systems take this idea further, and they look at whether you read right near the edge and use this to assume you're fetching a lot of data sequentially, so it will trigger prefetch for data afterwards. OSes like Linux are very complex in how they predict and prefetch files and clear the cache; I will likely take a simple approach similar to how Linux does it. 

My original idea was only to expose page pointers for processes to use however, I quickly realised this is a bad idea and diverting the problem and the much better way to handle it is to instead have a useful api like read range and read u64, etc., as this should be the goal of the file system driver to handle reading everything and which pages to use and what not. 

I was originally planning to have the data point to a page ID, then have the page system return the actual page location from that. However, I later decided it would be better to use a generic page data structure for different use cases, rather than having them all point to the page location. The reasoning here is that, instead of going page-by-page, you would always start with page data, which also lets you cheaply move the page if needed. It makes some things more complex, like finding all page cache files; however, it also makes other things simpler.

For reading files, I am planning to implement an async API that returns an ID, which can be used to check whether it can be read or what needs to be done to make it available. I might add a sync API; however, you should never really use sync in kernel space, as it wastes time doing nothing while it fetches data, not to mention making things messy and holding up info. Or, throughout this, while reading, 'async' is the wrong word to use; asis sync. It would yield, so I will use a sync API, but it will switch to async when the response is ready.

An interesting thing I found related to reading and writing is that you only have a single interrupt for read or write completions. This means that to know what you are reading or writing, you have to keep track of yourself on a list. It also can't know how many reads or writes have finished, as there is only 1 bit in the register for firing interrupts, meaning that 2 reads or writes finish before you handle the interrupt, and only one is fired. This means we have to keep a list of things to read or write and work our way through that list. Likely with a circular ring buffer; however, there would be real issues if you go over this limit, so I'm not quite sure about the approach yet. Could do a linked list, but I can imagine there would be a performance cost. NVMe tells you which has finished reading and writing, and keeps a list of them. I haven't looked into the specifics of it yet, but it is good to know. 

For the FS reading data I am thinking to do radix table of location to page used, for performance here I can make it so it will store each levels most left entry, idea is that if you want to store something but it is in first radix tree level it can just go directly to that tree level greatly speeding up fetching or if in 2nd can go to 2nd level. This means it can handle high-speed, low-RAM fetches for small files, but for large files, it still uses the same approach. Because of the exponential nature of radix tables, the approach would also work surprisingly well.

My idea here was to use a global ID for files and a similar idea for folders. It would greatly simplify storing all this data and handling cases where stuff relies on it. From what I have heard, this idea is used both in real OSes and internally for many filesystem formats, which makes a lot of sense, as it's much better than alternatives like a filesystem that uses full paths as IDs. I do see issues here with global ID conflicts between disks; however, for the time being, I will just use a global ID for each file on the OS internally, and later, when issues arise with this system, add a way for deduplication. I can see the approach of using an internal file ID different from a disk file ID, and of having internal save to a location and with a corresponding ID for the real file, which is likely an approach I will take later.
